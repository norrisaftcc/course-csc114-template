# CSC-114 Summer 2026 Pilot — 8-Week Crosswalk

**Pilot:** CSC-114 *Artificial Intelligence I*, 8-week Summer 2026.
**Cadence:** 2 meetings/week (Meeting A = concepts, Meeting B = studio/debugging) plus one mid-week "spike" — adopted from Manning's recommended rhythm.
**Spine:** Week 1 = legacy Cloud Agents (Module 0). Weeks 2–8 follow [`../textbook/manning_overview.md`](../textbook/manning_overview.md) week-by-week.
**Source material:** `planning/csc_dash/CSC-114/` and `planning/legacy_csc113/` are reusable assets, not the structural backbone. They get pulled *into* the Manning slots below.

The Canvas build uses **Learn → Practice → Apply → Assess**. Stakes are low while skills are being introduced (Weeks 1–3) and tighten as confidence grows (Weeks 5–8).

## What's cut from csc_dash and why

- **The first ~9 weeks of sklearn-first material** (Iris, Decision Trees, Random Forests, sklearn evaluation metrics as primary). Manning expects Keras `Sequential` by Week 2. Keeping sklearn-first would delay students' Keras fluency past Week 4, which Manning's Week 4–5 deliverables (universal workflow + project proposal) cannot tolerate. We *reference* sklearn for vocabulary (train/test split, accuracy, confusion matrix) but don't ramp it independently.
- **The full 6-sprint Scrum cycle.** csc_dash uses 2-week sprints with full ceremony (planning, daily standups, burndowns, reviews, retros). In an 8-week pilot, that's 4 sprints at best — and most pilot students won't have a CSC-113 lead-in. Per [[../../.claude/projects/-Users-norrisa-Documents-dev-github-course-csc114-template/memory/feedback_two_track_sacred_flow]], we split this by track: Code Builders use a **per-week mini-cadence** (issue → branch → PR → 1-paragraph retro); Prompt Masters submit by drop and skip the ceremonies entirely. The full Scrum cycle resurfaces in the capstone-2 sequence.
- **csc_dash Module 4 "AI Agents Sprint" (Weeks 7–8 in 16-week schedule).** Superseded by Week 1 Cloud Agents. The legacy version in `legacy_csc113/` is more developed and is the canonical agents content for this pilot.
- **csc_dash Modules 10–16 vision/NLP/capstone material** as *standalone* multi-week modules. Manning compresses vision (Ch 8–12), NLP (Ch 14–15), and generative (Ch 16–17) into Weeks 6–8 of the pilot. The csc_dash activities for transfer learning (M06) and NLP (M07) are still pulled in as individual lab assets.

## What's net new (no csc_dash equivalent)

- **Manning Ch 2 (Week 3): tensors, gradients, SGD/backprop.** csc_dash skips this — it's the math-for-debugging that classical-ML curricula don't need but neural-network curricula do. Use Manning's `chapter02_mathematical-building-blocks.ipynb` directly; the controlled-experiment write-up (LR or batch size) is the assessment.
- **Manning Ch 7 (Week 4): Keras deep dive.** csc_dash uses Keras but doesn't teach the Functional API, subclassing, or callbacks in depth. Sequential vs Functional is a real concept gap — plan studio time.
- **Universal workflow framing (Ch 6, Week 5).** csc_dash teaches sprints; Manning teaches "task framing → metric → evaluation protocol → baseline → overfit-then-regularize" as a craft. We're keeping Manning's framing because the project proposal artifact (1-page) is more pilot-appropriate than a sprint plan.

## Two-track PM expectations per week

Every week's Assess slot is delivered in two forms:

| Week | Code Builders submit | Prompt Masters submit |
|------|----------------------|----------------------|
| 1 | Files added to repo (GitHub Desktop OK). Screenshot of agent + 1-paragraph reflection. | Same. (No Sacred Flow yet — Week 1 is universal low-stakes.) |
| 2 | GitHub Issue → branch → PR with notebook + write-up. 1-paragraph retro inside the PR description. | Notebook + write-up dropped into repo via GitHub Desktop or web upload. |
| 3 | Same Sacred Flow as Week 2. Branch name encodes the experiment ("`week3-lr-sweep`"). | Drop. |
| 4 | PR includes the experiment log file. Retro mentions one change to the workflow itself. | Drop. |
| 5 | **First "tightened" stakes.** 1-page proposal in repo root, opened as a PR for review. Code Builders are expected to address reviewer comments before merge. | 1-page proposal dropped into repo. Optional reviewer comments. |
| 6 | PR per variant (train-from-scratch, feature-extract, fine-tune) OR one PR with each variant in its own commit. Whichever shows the comparison more clearly. | Drop. |
| 7 | PR with the project check-in artifacts. Retro names the one improvement iteration explicitly. | Drop. |
| 8 | Final PR merged to main of the project repo, tagged `v1.0`. Demo is a separate 5-minute deliverable (recorded video OK). | Drop into main branch, no tag required. Demo same as Code Builders. |

Both tracks earn identical points on the *artifact*. Code Builders earn **process credit** for the issue/PR/retro on top.
